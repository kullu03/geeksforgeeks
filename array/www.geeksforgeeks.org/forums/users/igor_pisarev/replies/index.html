    Simple but important observation here is the elements on L and R positions should be zeroes. Otherwise, we are flipping not the best segment.

    Thus, we should choose a segment that maximizes the value = (the number of 0’s) – (the number of 1’s).

    It seems we can do it in O(N): let’s store the index of the first 0 and the maximum difference that we have already found. If the current element is 0, check whether this obtained segment is better, update the L index and the maximum difference.

  • September 28, 2014 at 2:59 AM in reply to: Find Kth digit number after n changes #131535

    igor_pisarev
    Participant

    Hi! Look at the pattern of some first changes:

    i=0: 0
    i=1: 01
    i=2: 0110
    i=3: 01101001
    i=4: 0110100110010110
    i=5: 01101001100101101001011001101001

    As you can see, every time the number of digits doubles. Moreover, the first half of the new string is equal to the whole previous string; the second half is a mirrored previous string if the current change number is even, and is an inverted (negated) mirrored previous string if the current change number is odd. If you write this down, you should definetely see this pattern.

    Thus, we can solve this problem in O(N): first, calculate at which half K-th char is located in the final string (after N changes). This gives us whether we should consider the previous string normally or mirrored. Also, look at the parity of N – that tells us whether we should consider the previous string normally or inverted (negated). Perform this operation N times by going backwards from N to 1 and shrinking the length of the string every time twice. Finally, when you have come to the initial state 01, you can answer the question.

    By the way, what are the constraints on N and K? The length of the string number i is 2i-1, so if N > 64, the length of the string is greater than maximum value of int64 (unsigned long long).
    Also, the brute-force solution you have mentioned requires time and space proportional to the length of the final string which is O(2^n).

    Hi,

    let me define the probability of one valid combination (e.g. 01234) as P = (0.1)^5.
    Then look, which digits are valid for the first position.
    For ascending order: valid combinations starts with 0..5 (min is 01234, max is 56789).
    For descending order: valid combinations starts with 4..9 (min is 43210, max is 98765).

    Probability for ascending order is simply P * (number of valid combinations) which is 6*P.
    Probability for descending order is the same.

    Therefore, the resulting probability is 12*P which is 12 * (0.1)^5 = 0.00012

    Hi,

    I guess it’s pretty simple. First, build a suffix tree. It’s usually stored in such way that all the leaves represent the starting indices of this suffix in the initial string.
    Therefore, for any query, split it into list of words, then try to find all the words in the suffix tree by traversing it from top to bottom and storing the indices for each found word where it starts, as usual. If any of the words is not found, then return “invalid”. If all the words are found, check whether indices (in the same order as words in the query) form an increasing sequence. If yes, the answer is found, otherwise it’s invalid.

  • June 25, 2014 at 3:34 PM in reply to: Graph Theory #129870

    igor_pisarev
    Participant

    Hi,

    I came up with something like O(N^6) algorithm so although it’s polynomial (that’s what you asked) but it may be far from the most efficient algorithm for this problem.

    1) Reformulate problem in a following way: we need to find a subset of triangles where each of them shares the same node with exactly two triangles of the same subset. Then these triangles form a cycle by the sharing nodes with “sunflower” leaves which are distinct nodes (because they’re not shared with others).

    So, at first we should find all the triangles in the given graph. The upper bound of the number of triangles (if the given graph is complete) is equal to a binomial coefficient C(N, 3) which is proportional to N^3. The simplest way to find all the triangles is to brute force every triplet (i,j,k) which represent all the combinations of nodes form a triangle, and simply check whether this triangle is presented in graph or not.

    2) Construct another graph of triangles that were found in the previous step. Define triangles as nodes and put a link between any pair of them if they share some single node from the source graph.
    Now we are interested in determining whether a cycle exists in this graph such that no triangle in this cycle share more than one node with other triangles in the same cycle. I guess it can be done by DFS the triangles graph which can take up to O(<number of triangles> ^ 2) in the worst case so the overall complexity of this approach is proportional to O(N^6).

  • June 23, 2014 at 3:59 PM in reply to: Graph\'s and others #129850

    igor_pisarev
    Participant

    Hi,

    Since the number of nodes is small, we can brute-force all possible pairs (a,b) which are N*N in total. For every pair, we can detect if the source and the node i are in the same component. If not, this pair shouldn’t be counted.

    If yes, here comes the important observation: there is at most one cycle which the path from 1 (the source) to i comes through. This is true because from the input formatting we can see that each node is connected to only one another node.

    So now we can run DFS on the resulting graph to find a path from 1 to i. Note that this path may lie inside in cycle or not. If not, this pair should be counted if the length of this path is exactly K.

    The last case is if the path comes through the cycle (let me define the node starting this cycle as C, and the length of the cycle as L). Let me define the length of this path:

    D = dist(1, C) + dist(C, i)

    Note that we can run through the cycle as many times as we want, so node i can be reached from 1 in exactly K steps only if (K – D) is divisible by L (the remainder of the division is zero).

    This stage runs in N steps and needed for testing each pair, so the total time complexity is O(N^3).
