    there is a cute little trick to do this problem ,

    first extract all the start and the end characters of the strings – > linear time

    then form a graph as follows ,

    start in any String x if x has start and end characters as a and b then add and edge between a and b in the graph labeling it with the string number of string x , do this for all the strings -> linear time

    find an eulerian path / circuit in the graph that will be the chain – > linear time ( Fluery’s algorithm )
    if no path/ circuit then report as chain can’t be found .

    This might be a probable solution with o(1) space and o(n) time .

    In the process of merging the key invariant that has to be maintained is as follows .
    lets take the example

    A = [ 3 , 13 , 96 , 121 ] B= [ 7 , 18 , 45 , 78 , 111 ]

    let A’s pointer be a and B’s pointer be b
    initially a = 0 , and b = k+1
    1. 3 < 7 so a++
    2. 7 < 13 so b++
    if we had an external array then we would have put 3 and 7 in it without any problem , but now in this case we have to do it in place , so swap 7 and 13 ,
    now the problem is that the portion of A left to merge that is [ 13 , 96 , 121 ] has been distorted
    13 has gone past 96 and 121 .
    in order to solve this problem , we have to split the pointer a in 2 parts .
    let it be a1 and a2
    a1 points to to the start of the array left in A itself ,
    a2 points to the start of the pointer in B that has elements of A swapped into it .
    c marks the end of this portion in B , in c+1 the proper elements of B start .
    so when we take elements form left over portion of A to compare with b
    we move as follows ( a2 to c to a1 to end of A ) .

    let me know if i am wrong .

    The impossibility of the solution can be argued as follows

    in the Quick select algorithm , choosing a random pivot and then finding its position through the partition routine , the partition routine routine takes theta(n) time .

    similarly in the 2D version even if we choose a random pivot we have to find its position among n*n elements
    and its not possible to get that info without looking into all the elements , considering that all the elements are unordered .

    Lets say that we impose a condition that the rows are sorted , then we can find the item of ran K in nlogn time . as follows
    1. sort the rows and taking there max element .
    2. return the K smallest element .

    This can also be done in linear time . We have to run quick select on the max elements of the rows to get the ans .
