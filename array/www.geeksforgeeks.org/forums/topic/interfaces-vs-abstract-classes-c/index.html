    Abstract classes are to be used when there is a clear hierarchy, with common functionality and properties.

    Interfaces are used when a contract has to be enforced. The implementation of the contract may or may not be vastly different. The important thing is that the classes implementing the interface can be (and almost always are) vastly unrelated.

    For example, if you take the TextReader abstract class, it abstracts common functionality and properties for readers. These properties and methods would not make much sense for a non-reader class to inherit.

    On the other hand, an interface can be implemented by completely different and semantically unrelated classes. For example, your MyCustomReader and MyFavoriteGame can both implement IDisposable, which just means they are contractually obliged to provide the Dispose method.

    So yes, you can have a completely abstract abstract class which actually acts as an interface, but that is wrong from a design perspective.

    Abstract class and Interface are two different design decisions to be made according to our requirements. When we are planning to have identical functionality across our classes we can go ahead with Abstract classes. Abstract classes also makes our job easy when we expect component(say abstract layer) to change in future. Interfaces are preferred when we are looking for some kind of Contract between our components and outside world. It is also preferred when we need to implement common concise methods or functionality across similar classes. It is preferred to have segregated interfaces to avoid unnecessary & undesirable implementations on code side. Most of the times we do not get over the advantages of interfaces over abstract classes and vice versa.

    I prefer interfaces in most cases over abstract classes, though resolving the method would take a little extra time. Multiple interfaces can be implemented by a class thereby allowing to have different related functionality grouped yet made accessible only by(for) their type(interface in this case).
