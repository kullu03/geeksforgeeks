    Hey, is it written test question? or F2F interview? I like the post.

    See the explanation,

    In file1.c the identifier “arr” have array semantics. Where as in file2.c the semantics of “arr” is pointer type. The linker can’t resolve the name and at the same time won’t flag an error message. The program fails at runtime.

    The following correction can run without any error,

    int arr[10]; //in file1.c

    extern int arr[]; //in file2.c

    int main()

    {

    arr[0]=1;

    printf(“%d”, arr[0]);

    }

    I can give more concrete analysis after analyzing GNU output.

    @Dare, I don’t know the reason behind such behaviour. In the second file, the pointer is extern declaration. The linker may map it to an arbitrary address (it my guess). The behaviour is undefined.

    The key is, don’t use different semantics of same identifier.

    Some times, while working in collaboration with other teams, it is possible that two different programmers (or simply in two different files) can use same global variable name in two different files. In such case, if a third programmer (think it as third file) declares a variable at global level with same name, the linker atleast throw warning, and maps it to either of the two global definitions. The behaviour is undefined.

    @Venki : In first file, int arr[10] defined is global. So all its values( arr[0]….arr[9] ) are initialized with 0 by default. Whereas in second file, the same ‘arr’ declared is treated as pointer which makes it point to 0( the default value initialized to a[0] in first file).

    During linking, the second file searches for the declaration of ‘arr’ and finds int arr[10] in first file. It gets satisfied by this because ‘arr’ in first file is also a constant pointer(name of array is a constant pointer).
